The Benchmark Landscape

RL gyms, agent benchmarks, evaluation methodology — the full lay of the land

20 sources 99 claims extracted 25 adversarially verified 23 confirmed, 2 refuted
TL;DR

The landscape sits on two pillars: (1) the Farama Foundation ecosystem, which owns the de facto standard APIs for classical RL (Gymnasium + PettingZoo + 17 environment suites), and (2) a rapidly growing set of LLM agent benchmarks (AgentBench, WebArena, tau-bench, GAIA, SWE-bench) that test agents on realistic multi-step tasks.

A critical meta-finding from Princeton’s “AI Agents That Matter” paper: most benchmarks lack holdout sets (only 1 of 8 domain-general benchmarks had them), agents overfit to benchmark-specific policies, and accuracy-only evaluation ignores cost differences of up to 55×.

MuJoCo — The Physics Engine HIGH CONFIDENCE

MuJoCo is now open-source under Apache 2.0, maintained by Google DeepMind. It serves as the primary physics engine for continuous-control RL benchmarks—tasks like Ant, Humanoid, Walker2D, HalfCheetah.

It ships as a first-class Gymnasium environment suite, meaning you can pip install gymnasium[mujoco] and start training immediately.

Source: github.com/google-deepmind/mujoco

The Gymnasium API — How It Works

Gymnasium provides a unified Pythonic API with five core components:

  1. Environment interface — reset(), step(action), render()
  2. Registry — gymnasium.make("CartPole-v1")
  3. Space definitions — Box, Discrete, MultiBinary, etc.
  4. Wrappers — composable env transformations
  5. Vectorization — parallel environment execution

Bundled environment families: Classic Control (CartPole, Pendulum), Box2D (LunarLander, BipedalWalker), Toy Text, and MuJoCo.

Source: gymnasium.farama.org

Key Insight

DeepMind’s dm_env (401 stars) is the only alternative API, but adoption is negligible compared to Gymnasium. For multi-agent, PettingZoo is the closest to standard, though DeepMind’s Melting Pot uses its own interface—the multi-agent space remains more fragmented.

passk Explained

Most benchmarks report single-shot accuracy. tau-bench runs each task k times and reports passk—the probability of passing all k trials. This surfaces reliability issues that single-shot metrics hide. An agent that succeeds 70% of the time has a pass5 of only 17%.

Source: arxiv.org/abs/2406.12045 · GitHub

The Inverted Paradigm

GAIA deliberately inverts the typical benchmark difficulty curve: questions are easy for humans but hard for AI. This philosophy “departs from the current trend in AI benchmarks suggesting to target tasks that are ever more difficult for humans.”

What’s Broken & How to Fix It

The most important paper for anyone building a benchmark: “AI Agents That Matter” (Princeton, July 2024, 307 citations).

The Three Systemic Failures

1. Cost Blindness HIGH CONFIDENCE

Verified 3-0 by adversarial panel

Benchmarks only measure accuracy, ignoring cost. LATS (GPT-4) achieves 88% on HumanEval at $134.50. A simple baseline achieves 93.2% at $2.45—better accuracy at 55× less cost.

This means leaderboards reward expensive, over-engineered agents while simpler, cheaper approaches often perform better.

2. No Holdout Sets HIGH CONFIDENCE

Verified 3-0 by adversarial panel

Only 1 of 8 domain-general agent benchmarks surveyed had appropriate task holdouts. This enables agents to overfit to specific benchmark tasks.

Case study: STeP hardcodes policies for specific WebArena tasks (e.g., appending URL suffixes) that would fail on tasks from unseen websites.

3. Contamination Is Devastating

From benchmark contamination analysis papers

A 1.3B parameter model trained on leaked benchmark data can outperform models 10× its size (87.31% vs 26.49% on MMLU).

Three levels of contamination, from least to most harmful:

  1. Guideline contamination — exposure to annotation guidelines
  2. Raw text contamination — exposure to original text before annotation
  3. Annotation contamination — exposure to evaluation splits with labels (worst)

The 5-Dimension Evaluation Framework

From a 2025 ACL survey on LLM-based agent evaluation:

1. Data Curation

  • How tasks are sourced (real issues vs synthetic)

2. Environment

  • Static (fixed prompts) vs Dynamic (Docker containers, live systems)

3. Interface

  • Code / Tools / GUI

4. Metric Type

  • Accuracy, cost-aware, reliability (passk), functional correctness

5. Safety

  • Currently underdeveloped across the board
Caveat: Time Sensitivity

The AgentBench performance gap data uses pre-Llama-3 era open-source models (all under 70B params). The GAIA GPT-4 score of 15% is from late 2023. The holdout set survey covers benchmarks as of July 2024. All of these numbers have likely shifted—the structural insights about methodology remain valid.

Playbook: Building Your Own Benchmark

Based on the verified findings from 20 sources, here are the lessons for building a benchmark that actually measures something meaningful.

1. Have a Proper Holdout Set

Almost nobody does this—only 1 of 8 domain-general benchmarks had appropriate holdouts. Without holdouts, agents will overfit to your specific tasks and game the leaderboard. Split tasks into public dev / private test sets.

2. Measure Cost, Not Just Accuracy

Cost differences of up to 55× are common between agents with similar accuracy. Report cost-per-task alongside accuracy. Consider Pareto frontiers (accuracy vs cost) instead of single-number leaderboards.

3. Use Docker-Based Environments

WebArena’s model: each environment runs in its own Docker container, reset by restarting from the original image. This gives you reproducibility, contamination prevention, and full offline operation.

4. Use passk for Reliability

tau-bench’s innovation: run each task k times and report the probability of passing all k. A 70% single-shot agent has a pass5 of only 17%. This surfaces flakiness that single-shot metrics hide.

5. Test Rule Compliance, Not Just Task Completion

tau-bench’s key insight: agents deployed in production must follow domain-specific policies. Test whether agents can follow rules like refund policies, data handling procedures, escalation protocols—not just “did they complete the task.”

6. Evaluate End-State, Not Action Sequences

Compare database state / system state at the end of the task vs the annotated goal state (tau-bench) or check functional correctness of outputs (WebArena). Don’t match action-by-action—there are many valid paths to a correct outcome.

7. Plan for Evolution

Static benchmarks get contaminated and saturated. LiveBench refreshes monthly. Consider procedural generation, dynamic question variants, or time-sensitive questions sourced from recent data. At minimum, version your benchmark and track saturation curves.

Decision Checklist

Before finalizing your benchmark design, answer these questions:

  1. Is your evaluation static (fixed questions) or dynamic (interactive environments)?
  2. What’s your contamination prevention strategy?
  3. Do you have a proper holdout set?
  4. Are you measuring cost alongside accuracy?
  5. Are you measuring reliability (passk) or just single-shot performance?
  6. Are you testing task completion, rule compliance, or both?
  7. How will you evaluate—text matching, functional correctness, or end-state comparison?
  8. What’s your plan for benchmark evolution over time?

Open Research Gaps

Areas where the benchmark landscape is still thin:

  • Multi-modal agents — benchmarks for agents that see, hear, and interact with GUIs
  • OS-level / computer-use agents — beyond web navigation
  • Safety & policy compliance — tau-bench is pioneering but the field is underdeveloped
  • Private vs public evaluation suites — frontier labs use internal evals that differ significantly from public benchmarks
  • Cross-environment tasks — CRAB is first, but early
