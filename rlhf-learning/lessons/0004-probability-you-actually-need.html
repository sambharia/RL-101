SR1 (Lesson 3b) — Describe the attention mechanism in one sentence: what goes in, what comes out, what does it compute?

Each token projects its embedding into Query, Key, and Value vectors; dot products between every Query and Key produce attention weights (softmaxed to sum to 1); the output for each token is a weighted sum of all Value vectors — so each token's representation gets updated by the context it attends to most strongly.

SR2 (Lesson 3b) — What specific problem does the residual connection solve, and what is the mechanism?

Gradient vanishing in deep networks — gradients shrink exponentially as they flow back through many layers, so early layers stop learning. The residual adds 1 to the gradient: $\frac{\partial}{\partial x}(x + f(x)) = 1 + \frac{\partial f}{\partial x}$. The +1 creates a direct highway for gradients, independent of the layer's transformation.

SR3 (Lesson 3) — What does the loss $-\log P(y_{\text{correct}})$ equal when the model gives the correct token probability 0.05? Which direction does the gradient push?

$-\log(0.05) \approx 3.0$. The gradient for the correct token's weight is negative (increasing the weight decreases the loss), so the update rule $\theta \leftarrow \theta - \eta \nabla L$ increases the weight — pushing the model to assign more probability to the correct token.

1. Where we're headed

In a few lessons, you'll meet this equation — the heart of RLHF:

Preview: the RLHF objective (Lesson 7)
$$\max_{\pi}\; \mathbb{E}_{x \sim \mathcal{D},\, y \sim \pi(\cdot\,|\,x)}\!\Big[\,R(x,y)\,\Big] \;-\; \beta\,\text{KL}\!\big[\,\pi \;\|\; \pi_{\text{ref}}\,\big]$$

$\pi$ = a probability distribution (the policy). $\mathbb{E}$ = expectation. $y \sim \pi$ = sampling. $\text{KL}$ = built from log-probabilities.

Every symbol here is probability. Today we make you fluent in the four tools you need to read it.

You've already met pieces of this. Softmax gave you a probability distribution in Lesson 2. The loss $-\log P$ used log-probability in Lesson 3. Text generation sampled from a distribution. Today we formalize all of these and add the one big missing piece: expectation.

2. Probability distributions: assigning chances

You already know this intuitively — let's make it precise.

What you already know

In Lesson 2, softmax turned logits into numbers that sum to 1. The model said: "Given 'The cat sat on the', the probability of 'mat' is 0.35, 'floor' is 0.25, 'chair' is 0.15, …" That list of (outcome, probability) pairs is a probability distribution.

A probability distribution is just a complete answer to the question: "What could happen, and how likely is each thing?"

The two rules — that's all there is

A valid probability distribution $P$ over outcomes $\{x_1, x_2, \ldots, x_n\}$ must satisfy:

$$\text{Rule 1: } P(x_i) \geq 0 \quad \text{for every } i \qquad\qquad \text{Rule 2: } \sum_{i=1}^{n} P(x_i) = 1$$
Rule 1 No negative probabilities. A thing can't have a $-10\%$ chance of happening.
Rule 2 Something has to happen. The probabilities of all possible outcomes sum to exactly 1.

If a list of numbers follows both rules, it's a valid distribution. If it violates either, it's not.

Worked example: which of these are valid distributions?
DistributionValuesValid?
A[0.4, 0.3, 0.3]Yes — all ≥ 0, sum = 1.0
B[0.5, 0.6, -0.1]No — has a negative value
C[0.2, 0.3, 0.4]No — sum = 0.9, not 1.0
D[1.0, 0.0, 0.0]Yes — one outcome is certain, the rest impossible

Distribution D is interesting: it says "outcome 1 happens with 100% certainty." Zeros are fine. Negative numbers aren't. And the sum must hit exactly 1.

Why softmax guarantees a valid distribution

In Lesson 2, you saw softmax: $P(x_i) = \frac{e^{z_i}}{\sum_j e^{z_j}}$. Now you can see why it works: $e^{z_i}$ is always positive (Rule 1 satisfied), and dividing by the sum of all $e^{z_j}$ forces the result to sum to 1 (Rule 2 satisfied). Softmax is a machine that turns any list of numbers into a valid probability distribution.

Notation: $P(x)$ and $\pi(y \mid x)$

Two notations you'll see constantly:

Reading probability notation
$P(\text{mat})$ "The probability of 'mat'." One number between 0 and 1.
$\pi(y \mid x)$ "The probability of output $y$ given input $x$." The vertical bar $|$ means "given" or "conditioned on." This is what a language model computes: $P(\text{next token} \mid \text{all previous tokens})$.
$\pi(\cdot \mid x)$ "The whole distribution over all possible $y$, given $x$." The dot means "all possible values." This is the entire softmax output — a list of 50,257 probabilities.

The letter $\pi$ (pi) is used instead of $P$ when we want to emphasize that this distribution is a policy — an agent's strategy for choosing actions. A language model's next-token distribution is a policy: given the context $x$, it chooses an action $y$ (the next token).

Worked example: reading the notation

A model sees the context "The cat sat on the" and produces:

$\pi(\text{mat} \mid \text{"The cat sat on the"}) = 0.35$. The whole distribution $\pi(\cdot \mid \text{"The cat sat on the"})$ is the full column of 50,257 numbers.

3. Expectation: the weighted average

Now for the concept that matters most for RLHF. It answers a simple question:

The question expectation answers

If I repeat a random process many times, what will the average outcome be?

Or more precisely: given a probability distribution, what value do I get "on average"?

The problem: judging a policy

Suppose you have two language models, and you want to know which one gives better responses. Each model could produce different responses to the same prompt — some great, some mediocre, some bad. You can't just look at one response. You need to ask: "On average, how good are this model's responses?"

That "on average" is expectation.

Worked example: expected reward of a policy

Imagine a model can produce 3 possible responses to "What is 2+2?", with these probabilities and reward scores:

Response $y$Probability $\pi(y)$Reward $R(y)$
"4"0.710
"four"0.28
"5"0.10
What's the expected reward?
Each outcome contributes: (how likely it is) × (how good it is).

$\mathbb{E}[R] = 0.7 \times 10 + 0.2 \times 8 + 0.1 \times 0$
$= 7.0 + 1.6 + 0.0$
$= \mathbf{8.6}$

"4" contributes 7.0 to the average (high probability × high reward). "5" contributes 0.0 (low probability × zero reward). The expected reward of this policy is 8.6.

The key insight

Expectation is a weighted average where the weights are probabilities. An outcome that's likely matters more than one that's rare. An outcome that's impossible (probability 0) contributes nothing, no matter how good or bad it is.

This is exactly why RLHF maximizes expected reward: you don't just want the model to occasionally produce a great answer. You want the answers it typically produces to be good.

The expectation formula
$$\mathbb{E}_{y \sim \pi}[f(y)] = \sum_{y} \pi(y) \cdot f(y)$$
$\mathbb{E}$ "Expected value of" — the weighted average.
$y \sim \pi$ "$y$ drawn from distribution $\pi$" — the subscript tells you which distribution the probabilities come from.
$f(y)$ Some function of the outcome — in RLHF, this is the reward $R(y)$.
$\sum_y$ Sum over all possible outcomes.

Read it as: "For each possible outcome $y$, multiply how likely it is $\pi(y)$ by how valuable it is $f(y)$, and add them all up."

Worked example: comparing two policies

Two models answer "What is the capital of France?"

Model A:

Response$\pi_A(y)$$R(y)$$\pi_A(y) \cdot R(y)$
"Paris"0.8108.0
"Lyon"0.1520.3
"London"0.0500.0

$\mathbb{E}_A[R] = 8.0 + 0.3 + 0.0 = \mathbf{8.3}$

Model A has higher expected reward (8.3 vs 5.6). Both models can produce the right answer, but A does it more reliably. RLHF's goal is to push the policy from B-like to A-like — increase the probability of high-reward responses.

Why "expected" doesn't mean "what you expect"

The expected value of a fair die roll is 3.5. You will never roll a 3.5. The expected value isn't a prediction of any single outcome — it's the average if you roll many times. In RLHF, the expected reward isn't what one particular response will score. It's how good the model's responses are on average, weighted by how likely the model is to produce each one.

4. Sampling: drawing from a distribution

You've seen sampling before — in Lesson 2, the generation loop picked the next token by "rolling a weighted die." Let's formalize this and understand why sampling is the only practical option for RLHF.

The problem: expectations are impossible to compute exactly

Why we can't just sum

The expectation formula says: sum over all possible outcomes. For our toy examples with 3 responses, that's easy. But in RLHF, a "response" is a sequence of maybe 500 tokens, each chosen from 50,257 options. The number of possible responses is roughly $50{,}257^{500}$ — a number with about 2,350 digits. You cannot sum over that. The universe doesn't have enough time.

So instead of computing the exact expected reward, you approximate it. How? Generate a few actual responses, score them, and average the scores. That's sampling.

What sampling means

$y \sim \pi(\cdot \mid x)$ is read: "draw $y$ from the distribution $\pi$, given input $x$."

Mechanically: generate a random number, use the distribution's probabilities to decide which outcome that random number corresponds to.

$$\text{If } \pi = [0.7, 0.2, 0.1], \text{ then } \begin{cases} y = \text{"4"} & \text{70% of the time} \\ y = \text{"four"} & \text{20% of the time} \\ y = \text{"5"} & \text{10% of the time} \end{cases}$$
Worked example: estimating expectation by sampling

True expected reward: $\mathbb{E}[R] = 0.7 \times 10 + 0.2 \times 8 + 0.1 \times 0 = 8.6$

Now suppose we draw 5 samples from $\pi = [0.7, 0.2, 0.1]$:

Sample #Drawn $y$$R(y)$
1"4"10
2"four"8
3"4"10
4"4"10
5"5"0
Sample average:
$\hat{\mathbb{E}}[R] = \frac{10 + 8 + 10 + 10 + 0}{5} = \frac{38}{5} = \mathbf{7.6}$

Not exactly 8.6 — but close. With 5 samples, we're off by about 1. With 100 samples, we'd be much closer. With 1,000, nearly exact. The more samples, the better the estimate. This is the law of large numbers in action.

Monte Carlo estimation

Draw $N$ samples $y_1, y_2, \ldots, y_N$ from $\pi$. The sample average approximates the true expectation:

$$\mathbb{E}_{y \sim \pi}[f(y)] \;\approx\; \frac{1}{N} \sum_{i=1}^{N} f(y_i)$$

This approximation improves as $N$ grows. It has a name: Monte Carlo estimation. It's how nearly all RL algorithms work in practice — including the ones that train ChatGPT.

Sampling in text generation — you've done this

In Lesson 2, the generation loop was: run gpt() → softmax → sample one token from the distribution → append → repeat. Each "sample one token" step is exactly $y \sim \pi(\cdot \mid x)$. Text generation is sampling from a policy. Now you know the formal name for what you already understood.

Why RLHF can't avoid sampling

The RLHF training loop goes: sample a prompt → sample a response from the current model → score it with the reward model → update the policy. Every training step requires sampling an actual response. You can't compute the expected reward over all possible responses, because there are astronomically many. So you sample a few, compute their rewards, and use the average as your estimate. This is why RL training is inherently noisy — and why you need many training steps to converge.

5. Log-probabilities: why $\log P$ is everywhere

You've already used $-\log P$ as the loss function in Lesson 3. But log-probabilities appear everywhere in RLHF — in the objective, in KL divergence, in policy gradients. Let's understand why.

The problem with raw probabilities

When numbers get too small

A language model produces one token at a time. The probability of a 500-token response is the product of 500 individual token probabilities:

$P(\text{response}) = P(y_1) \times P(y_2) \times \cdots \times P(y_{500})$

If each token has probability around 0.1, the response probability is $0.1^{500} = 10^{-500}$. That's a number with 500 zeros after the decimal point. No computer can store it — it underflows to exactly 0. All your math breaks.

The fix: use logarithms. Instead of multiplying tiny probabilities, you add their logs.

Three properties that make logs essential

Here's why ML uses $\log P$ instead of $P$:

Property 1: products become sums
The log rule:
$\log(a \times b) = \log a + \log b$
Applied to a 500-token response:
$\log P(\text{response}) = \log P(y_1) + \log P(y_2) + \cdots + \log P(y_{500})$

Instead of multiplying 500 numbers (which underflows), we add 500 numbers (which is safe). If each $\log P(y_i) \approx -2.3$ (i.e., $P \approx 0.1$), then $\log P(\text{response}) \approx -1150$. A large negative number, but a perfectly fine one for a computer to work with.

Property 2: log is a monotonic transformation
If $P(A) > P(B)$, then $\log P(A) > \log P(B)$.
The ranking is preserved. The most probable outcome still has the highest log-probability.

So maximizing $P$ is the same as maximizing $\log P$. We can work with log-probabilities everywhere and never lose any information about which outcome is most likely.

Property 3: nice gradient behavior
The derivative of $\log P$ with respect to $P$:
$\frac{d}{dP} \log P = \frac{1}{P}$

When $P$ is small (say 0.01), the gradient $1/P = 100$ — a big push. When $P$ is already large (say 0.99), the gradient $1/P \approx 1$ — a small push. This is exactly what we want: if the model barely assigns any probability to the correct token, the gradient is huge ("fix this!"). If the model already gets it right, the gradient is small ("you're fine, move on").

The log scale: building intuition

$P$$\log P$ (natural log)$-\log P$ (cross-entropy loss)Intuition
1.00.00.0Certain — zero surprise, zero loss
0.5−0.690.69A coin flip — moderate surprise
0.1−2.302.30Unlikely — substantial surprise
0.05−3.003.00This was the loss in Lesson 3!
0.01−4.614.61Very unlikely — high surprise
0.001−6.916.91Almost impossible — extreme surprise

Notice: as $P$ drops by a factor of 10, $\log P$ drops by about 2.3. The log compresses the scale. A probability 1000× smaller is only a log-probability about 7 units smaller. This compression is why computers can work with log-probs when raw probabilities would underflow to zero.

Log-probability notation you'll see
$\log \pi(y \mid x)$ "The log of the probability that policy $\pi$ assigns to response $y$ given prompt $x$." Always negative (since $0 < \pi \leq 1$, and log of a number ≤ 1 is ≤ 0).
$-\log \pi(y \mid x)$ The cross-entropy loss — always positive. You already know this: higher = worse. This is the loss from Lesson 3, now with policy notation.
$\log \frac{\pi(y|x)}{\pi_{\text{ref}}(y|x)}$ The log ratio of two policies. Positive if $\pi$ gives $y$ higher probability than $\pi_{\text{ref}}$. This ratio is the building block of KL divergence (coming in a later lesson).
Worked example: log-ratio of two policies

Policy $\pi$ gives "Paris" probability 0.8. Reference policy $\pi_\text{ref}$ gives "Paris" probability 0.5.

Log-ratio:
$\log \frac{\pi(\text{Paris})}{\pi_\text{ref}(\text{Paris})} = \log \frac{0.8}{0.5} = \log 1.6 \approx 0.47$

Positive — the new policy is more confident about "Paris" than the reference. In RLHF, a large positive log-ratio means the model has drifted far from its starting point for this output. The KL penalty term keeps this in check.

What if the new policy gives "Paris" only 0.3?
$\log \frac{0.3}{0.5} = \log 0.6 \approx -0.51$

Negative — the new policy is less confident about "Paris" than the reference. The model has shifted probability away from this token.

The connection: cross-entropy loss IS log-probability

In Lesson 3, you learned the loss $-\log P(y_\text{correct})$. Now you can see what this really is: the negative log-probability of the correct token under the model's distribution. Minimizing this loss is the same as maximizing the log-probability of the correct answer — which is the same as maximizing the probability. All three statements are mathematically equivalent. You've been working with log-probabilities since Lesson 3 without knowing it had a name.

6. Reading the equation

Now let's go back to the preview and read it with your new vocabulary:

The RLHF objective — now you can read it
$$\max_{\pi}\; \underbrace{\mathbb{E}_{x \sim \mathcal{D},\, y \sim \pi(\cdot\,|\,x)}}_{\text{sample a prompt, sample a response}} \!\Big[\;\underbrace{R(x,y)}_{\text{reward}}\;\Big] \;-\; \underbrace{\beta\,\text{KL}\!\big[\,\pi \;\|\; \pi_{\text{ref}}\,\big]}_{\text{don't drift too far (uses log-ratios)}}$$

Reading it left to right:

$\max_\pi$ — Find the policy (probability distribution over responses) that maximizes…

$\mathbb{E}_{x \sim \mathcal{D}, y \sim \pi}$ — …the expected value, where prompts $x$ are drawn from a dataset $\mathcal{D}$ and responses $y$ are sampled from the policy $\pi$, of…

$R(x,y)$ — …the reward (how good is this response to this prompt)…

$- \beta \text{KL}[\pi \| \pi_\text{ref}]$ — …minus a penalty that grows when the policy's log-probabilities drift too far from a reference policy's log-probabilities.

Every symbol in this equation uses probability: $\pi$ is a distribution, $\mathbb{E}$ is expectation, $\sim$ is sampling, and KL is built from log-ratios. You now have all four tools.

The plain-English translation

"Find a language model that, on average across many prompts and sampled responses, gets high reward scores — but don't let it become too different from where it started."

Every word in this sentence maps to a symbol above. You can now read both versions.

Low-stakes warm-up — just for you, not the gate.

Q1 — A distribution $P$ assigns probabilities [0.4, 0.35, 0.25] to outcomes with values [2, 6, 10]. What is $\mathbb{E}[X]$?

Correct. $0.4 \times 2 + 0.35 \times 6 + 0.25 \times 10 = 0.8 + 2.1 + 2.5 = 5.4$. Notice this is NOT the simple average of the values (which would be 6.0). The probabilities act as weights — outcome "2" is most likely, so it pulls the average below 6.

The simple average of {2, 6, 10} is 6.0, but expectation is a weighted average. Each value is weighted by its probability: $0.4 \times 2 + 0.35 \times 6 + 0.25 \times 10 = 5.4$.

Q2 — Why do ML systems use $\log P$ instead of $P$ when computing sequence probabilities?

Correct. A 500-token sequence with per-token $P \approx 0.1$ gives $P(\text{seq}) = 0.1^{500} = 10^{-500}$, which is computationally zero. But $\log P(\text{seq}) = 500 \times (-2.3) = -1150$ — a perfectly representable number. Products become sums under log, and sums don't underflow.

The core reason: multiplying many small probabilities (each < 1) produces a number so tiny the computer rounds it to 0. Taking the log converts multiplication to addition — $\log(a \times b) = \log a + \log b$ — and a sum of manageable negative numbers never underflows.

Q3 (hard) — You sample 10 responses from a policy. 8 receive reward 10, 2 receive reward 0. Your sample estimate of $\mathbb{E}[R]$ is 8.0. But the true $\mathbb{E}[R]$ is 8.6. What is the most precise explanation of this gap?

Correct. Monte Carlo estimation is unbiased — on average, the sample mean equals the true expectation. But any finite sample has variance (random noise). With 10 samples, the estimate could land above or below 8.6. With 10,000 samples, it would be almost exactly 8.6. The gap isn't bias — it's variance from a small sample size.

Monte Carlo estimation is unbiased — the expected value of the sample average IS the true expectation. The gap between 8.0 and 8.6 is sampling variance (random noise from a small $N$), not systematic error. As $N \to \infty$, the sample average converges to 8.6 with probability 1 (law of large numbers).

SR1 — What is the formula for expectation, and what does each symbol mean? Give a one-sentence plain-English translation.

$\mathbb{E}_{y \sim \pi}[f(y)] = \sum_y \pi(y) \cdot f(y)$. "For each possible outcome, multiply how likely it is by how valuable it is, and add them all up." It's a weighted average where the weights are probabilities.

SR2 — Why does RLHF training use sampling instead of computing the exact expectation? What is the fundamental barrier?

The space of all possible responses is astronomically large ($\sim 50{,}257^{500}$). Summing over every possible response to compute the exact expectation is computationally impossible. Instead, you sample a few responses, score them, and average — a Monte Carlo estimate that converges to the true expectation as the number of samples grows.

SR3 — What are the three key properties of logarithms that make log-probabilities useful in ML?

(1) Products become sums: $\log(ab) = \log a + \log b$ — prevents underflow when multiplying many probabilities. (2) Monotonic: if $P(A) > P(B)$ then $\log P(A) > \log P(B)$ — rankings are preserved, so maximizing $\log P$ is the same as maximizing $P$. (3) Gradient scales inversely: $\frac{d}{dP}\log P = 1/P$ — large gradient for low-probability events (strong learning signal where it's needed most).
