The Three-Step RLHF Pipeline

Pre-training gives you a base model that predicts text. Post-training turns it into an assistant. Three steps: SFT, Reward Model, RL. After this lesson, you know the full map.

Phase 1 — The RLHF big picture Prerequisites: All of Phase 0 (Lessons 1–5b) ~30 min

Warm-up: pull it back from memory

SR1 (Lesson 5) — What are the six core RL terms? State the RL objective in one sentence.

Agent, state, action, reward, policy, return. RL objective: find the policy that maximizes expected return.

SR2 (Lesson 5b) — In your capstone, what did you identify as the part of the training loop that changes when you go from supervised to RLHF?

The loss signal changes. In supervised training, the loss compares against the correct next token from the data. In RLHF, the loss comes from a reward model's score. The forward pass (gpt function) stays the same.

SR3 (Lesson 4) — What is the RLHF objective in symbols? Name each part.

$\max_\pi \mathbb{E}_{y \sim \pi}[R(y)] - \beta \text{KL}[\pi \| \pi_\text{ref}]$. Maximize expected reward minus a penalty for drifting too far from the reference policy.

1. From raw text to helpful assistant

A language model starts life by reading the internet. It learns to predict the next token — that's all. If you give it a question, it might continue it with another question, or with a Wikipedia paragraph, or with nonsense. It has knowledge but no behavior.

The journey from "predicts text" to "helpful assistant" has two big phases:

PRE-TRAINING POST-TRAINING Read the internet Turn into an assistant Learn to predict text ───▸ Learn to be helpful (Lessons 2–3b) (This lesson onward) Three steps: Step 1: SFT Step 2: Reward Model Step 3: RL (RLHF)

Pre-training is what microGPT does: predict the next token from data. You already understand this completely. Post-training is where RLHF lives. It has three steps, each building on the last.

2. The three steps

Here's the full pipeline at a glance. Then we'll zoom into each step.

Base Model ──SFT──▸ SFT Model ──Train RM──▸ Reward Model │ SFT Model ───────── RL + Reward Model ─────────▸ Final Model

3. Step 1: Supervised Fine-Tuning (SFT)

The base model predicts text, but it doesn't know how to answer questions. It's never seen a conversation format. SFT teaches it the format.

What SFT actually does

You take the base model and train it on a dataset of (prompt, ideal response) pairs — written by humans. The training is exactly the same as what you did in microGPT: next-token prediction with cross-entropy loss. The only difference is the data.

Worked example: SFT training data
Pre-training data (raw internet):
"The capital of France is Paris. Paris is located on the Seine river and has a population of..."
SFT data (conversations):
User: What is the capital of France?
Assistant: The capital of France is Paris.

Same training method ($-\log P$ of the correct next token), different data format. The model learns: when you see "User: [question]", respond with "Assistant: [helpful answer]".

SFT training — you already know this

SFT loss is the same cross-entropy loss from Lesson 3:

$$\mathcal{L}_\text{SFT} = -\sum_{t} \log \pi_\theta(y_t \mid x, y_{$y_t$ The correct token at position $t$ (from the human-written response).
$y_{ All tokens before position $t$.
$x$ The prompt.

This is exactly -probs[target_id].log() from microGPT — summed over every token in the response.

What SFT can and can't do

Can do: Teach the model the conversation format. Make it answer questions instead of continuing text. Give it a "voice" — helpful, concise, structured.

Can't do: Make the model choose the best answer among many good ones. SFT treats every human-written response as equally correct. It can't learn "this response is better than that one" — only "this is what you should say." For preference and quality, you need Steps 2 and 3.

What comes out of SFT

An SFT model — a language model that can have conversations and follow instructions. It's already useful. But it's not great at subtle quality judgments: when is a longer answer better? When should it refuse? How formal should it be? Those need human preferences.

4. Step 2: Train a Reward Model

The SFT model can answer questions. Now we need a way to measure how good each answer is — automatically, without a human scoring every response. That's the reward model.

How you build one

a. Collect comparisons
Give a prompt to the SFT model. Generate two (or more) different responses. Show both to a human and ask: "Which response is better?" The human picks one. Repeat thousands of times.
b. Build a dataset of preferences
Each data point looks like: (prompt, winning response, losing response). No scores — just "A is better than B."
c. Train a model to predict the human's preference
Take a language model (often the SFT model itself), replace the final layer so it outputs one number instead of token probabilities. Train it so that: winning response gets a higher score than losing response.
Worked example: reward model training data

Prompt: "Explain gravity to a 5-year-old."

Response A: "Gravity is the force of attraction between objects with mass, proportional to the product of their masses and inversely proportional to the square of the distance between them."

Response B: "You know how when you throw a ball up, it comes back down? That's gravity! The Earth is really big and heavy, so it pulls everything toward it — like a giant magnet for everything."

Human judgment: B is better (appropriate for a 5-year-old).

After training:
Reward model scores: R(A) = 3.2, R(B) = 7.8
Higher score for B — the reward model learned the human's preference.
Reward model architecture

The reward model is a transformer — same architecture as the language model — but with a different final layer:

Language model Input tokens → transformer → logits over vocabulary (50,257 numbers)
Reward model Input tokens → transformer → single scalar score (1 number)

Think of it as: replace lm_head [27×16] from microGPT with a reward_head [1×16]. Instead of 27 logits, you get 1 number: the reward.

Why not just ask humans every time?

During RL training (Step 3), the model generates thousands of responses per batch. You can't have humans score each one — it's too slow and expensive. The reward model is a proxy for human judgment: it learned what humans prefer from the comparison data, and now it can score new responses automatically. It's not perfect, but it's fast enough to run inside a training loop.

The reward model is also a language model

This surprises people. The reward model is typically initialized from the SFT model — same transformer, same weights — with just the output layer swapped. It reads the prompt and response, processes them through the same attention and MLP blocks, and then produces a single number instead of next-token probabilities. It "understands" language the same way the policy does — it just outputs a score instead of text.

5. Step 3: RL optimization (the RLHF step)

Now you have everything you need:

  • A policy (the SFT model) that can generate responses
  • A reward model that can score them

Step 3 connects them: generate responses, score them with the reward model, and update the policy to produce higher-scoring responses.

a. Sample a prompt from the dataset
"Explain photosynthesis."
b. Generate a response using the current policy
The model (agent) samples tokens one at a time: "Plants convert sunlight into energy through..."
c. Score the response with the reward model
Reward model reads prompt + response and outputs a score: R = 7.2
d. Update the policy to increase the probability of high-reward responses
Use policy gradient methods (coming in Lessons 10–11) to push the model toward generating responses like this one more often — while staying close to the original SFT model (the KL penalty).
The RLHF objective — now you know every piece
$$\max_{\pi}\; \mathbb{E}_{x \sim \mathcal{D},\, y \sim \pi(\cdot\,|\,x)}\!\Big[\,R(x,y)\,\Big] \;-\; \beta\,\text{KL}\!\big[\,\pi \;\|\; \pi_{\text{ref}}\,\big]$$
$\pi$ The policy being trained (started as the SFT model).
$\pi_\text{ref}$ The reference policy (the SFT model, frozen — never updated).
$R(x,y)$ The reward model's score for response $y$ to prompt $x$.
$\text{KL}[\pi \| \pi_\text{ref}]$ How far the policy has drifted from the reference. Built from log-ratios: $\log \frac{\pi(y)}{\pi_\text{ref}(y)}$.
$\beta$ Controls the tradeoff: high $\beta$ = stay close to SFT model, low $\beta$ = chase reward harder.
Why the KL penalty?

Without the KL penalty, the model would find degenerate ways to maximize the reward score — repeating phrases the reward model likes, producing nonsensical text that happens to score well, or collapsing to one memorized response. The KL penalty says: "you can improve, but don't become a completely different model." It keeps the policy close to the SFT model's natural language abilities while allowing targeted improvements.

Worked example: one RLHF training step
Prompt:
"What's the best way to learn programming?"
Policy generates:
"Start with Python. Build small projects. Read other people's code."
Reward model scores:
R = 8.1 (high — helpful, concise, actionable)
Log-ratio check:
$\log \frac{\pi(\text{response})}{\pi_\text{ref}(\text{response})} = 0.3$ (small drift — fine)
Update:
Increase probability of responses like this one. The KL is small, so the update goes through without much penalty.

Now suppose the model generates: "PYTHON PYTHON PYTHON BEST BEST BEST"

Reward model scores:
R = 6.5 (the word "Python" helped, but it's garbage)
Log-ratio check:
$\log \frac{\pi(\text{response})}{\pi_\text{ref}(\text{response})} = 4.7$ (huge drift — the SFT model would never say this)
Effective objective:
$8.1 - \beta \times 4.7$ — the KL penalty eats the reward. Net score is low. Model does NOT learn to repeat this.

6. Putting it all together

Step 1: SFT Base model + conversation data SFT model $-\log P(\text{next token})$ on conversations Same method, new data
Step 2: Reward Model SFT model + human comparisons Reward model (outputs 1 number) Winning response should score higher (New — this lesson)

The plain-English summary

Pre-training: Read the internet, learn to predict text.
SFT: Read conversations, learn to answer questions.
Reward model: Learn what humans prefer by looking at comparisons.
RL: Generate responses, score them, get better at generating high-scoring ones — but don't go crazy.

7. The post-training landscape (beyond the three steps)

The three-step pipeline (SFT → RM → RL) is the classic RLHF recipe from the original InstructGPT paper (2022). But the field has evolved. Here's what the landscape looks like now:

Variations on the theme
DPO (Direct Preference Optimization):
Skip the reward model entirely. Train directly on preference pairs using a clever loss function that implicitly defines the reward. Simpler pipeline, fewer moving parts. (Lesson 12)
RLAIF (RL from AI Feedback):
Replace human comparisons with AI-generated comparisons. A strong model judges which response is better instead of a human. Cheaper and faster to scale. (Lesson 17)
Rejection Sampling:
Generate many responses, score them all with the reward model, and fine-tune on only the best ones. No RL algorithm needed — just cherry-pick and train with SFT. The simplest "RLHF" that works. (Lesson 13)
RLVR (RL with Verifiable Rewards):
For math and code, you don't need a reward model — you can check if the answer is correct automatically. The reward is binary: right or wrong. (Lesson 14)

All of these are variations on the same core idea: get a signal for what's good, and push the model toward it. The classic three-step pipeline is the foundation that makes all the variations understandable.

Self-check

Q1 — What is the key difference between pre-training and SFT?

Correct. SFT is mechanically identical to pre-training — same loss function, same training loop. The only difference is the data: pre-training uses raw text, SFT uses curated (prompt, response) pairs. The format teaches the model to behave as an assistant.

SFT uses the exact same training method as pre-training — predict the next token, compute $-\log P(\text{correct})$. The only change is the data: instead of raw internet text, you train on curated (prompt, response) conversations.

Q2 — What does the reward model output?

Correct. The reward model reads the prompt and response, processes them through a transformer, and outputs a single number. Higher = better. Think of it as replacing lm_head [27×16] with reward_head [1×16].

The reward model is a transformer that outputs a single scalar score — one number. Not probabilities over tokens, not corrected text, not binary. Just one number: how good is this response?

Q3 (hard) — Why does RLHF need a KL penalty? What would go wrong without it?

Correct. The reward model is an imperfect proxy for human judgment. Without the KL penalty, the policy would exploit the reward model's weaknesses — producing text that scores high but is actually low quality. The KL penalty constrains the policy to stay close to the SFT model, which already generates natural, coherent language. This is a form of Goodhart's Law: "when a measure becomes a target, it ceases to be a good measure."

Without the KL penalty, the model would "hack" the reward model — finding degenerate text that scores high but isn't actually good (repetition, strange phrasing, exploiting blind spots). The KL penalty keeps the model close to the SFT model's natural language, preventing this reward hacking.

For next time: questions to answer from memory

SR1 — What are the three steps of the RLHF pipeline? What is the training signal for each?

(1) SFT: train on (prompt, response) pairs using cross-entropy loss ($-\log P$ of correct token). (2) Reward Model: train on human preference comparisons — winning response should score higher. (3) RL: generate responses, score with reward model, maximize $\mathbb{E}[R] - \beta \text{KL}$.

SR2 — What does the reward model output, and how is its architecture related to the language model?

The reward model outputs a single scalar score (one number). Its architecture is the same transformer as the language model, but with the final layer replaced: instead of projecting to vocab_size logits, it projects to 1 number. Often initialized from the SFT model's weights.

SR3 — Why does RLHF need the KL penalty? What goes wrong without it?

Without KL, the policy exploits the reward model — producing degenerate text (repetition, nonsense) that scores high but isn't actually good. The KL penalty keeps the policy close to the SFT model's natural language, preventing reward hacking. It's Goodhart's Law in action.
