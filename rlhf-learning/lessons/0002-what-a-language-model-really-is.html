Q1 — Name the three post-training stages in order, and state the data type each needs.

SFT — demonstration data: human-written example completions.

Reward model training — comparison pairs: two answers to the same prompt, human picks the better one.

RL optimisation — the policy generates completions; the reward model scores them; gradient descent moves the policy toward higher scores.

Q2 — Why does RLHF need SFT first? What breaks if you skip it?

A base model rambles — it continues training text, not answers. A reward model needs two coherent completions to compare. If neither makes sense, the preference signal is noise. SFT instills the Q&A format first so there is something meaningful to rank.

1. Start here: the exact algorithm ChatGPT runs

Everything in this lesson is an explanation of this one block of code:

tokens = tokenize(prompt)          # break text into integer IDs

while True:
    logits = transformer(tokens)   # neural net → one score per vocabulary entry
    probs  = softmax(logits)       # scores → probabilities that sum to 1
    next   = sample(probs)         # draw one token from that distribution
    tokens.append(next)            # grow the context, feed it back in
    if next == END_TOKEN:
        break

response = detokenize(tokens)      # integers → readable text

This loop runs once per output token. A 200-word response is 200 passes through this loop. There is nothing else. Each section below explains one line.

What is transformer(tokens) physically?

A Transformer is a large neural network — for GPT-4, roughly 96 layers deep — that reads all the tokens at once using a mechanism called attention (letting each token "look at" every other token) and outputs a score for every token in the vocabulary. We'll go deep on the architecture in a later lesson. For now: treat transformer(tokens) as a black box that outputs one raw number per vocabulary entry. Those numbers are called logits. The word comes from "log-odds" — they can be any real number, positive or negative.

2. tokenize(prompt) — what is a token?

A language model does not see characters, and it does not see words. It sees tokens — subword chunks produced by a fixed compression algorithm that runs before the model ever starts. Here is the phrase "RLHF is surprisingly powerful" as a modern tokenizer sees it:

RL#9987 HF#13909  is#374  sur#15382 pris#29473 ingly#4315  powerful#6787

7 tokens total. Each chip = one token. The small ID is its integer index in the vocabulary. Notice: "RLHF" splits into [RL][HF]; the space before "is" is part of the " is" token.

After tokenisation, the prompt is just a Python list of integers: [9987, 13909, 374, 15382, 29473, 4315, 6787]. That integer list is what gets passed into the neural network.

The vocabulary is the fixed master list of all possible tokens. GPT-2 had ~50,000; modern models use 100,000–200,000. Every token the model reads or writes must be in this list. "Tokenize" encodes text into IDs; "detokenize" reverses it.

Why subwords — not characters or words?

Characters make sequences very long (expensive) and give the model no useful structure to learn from. Whole words would need millions of entries to cover all morphological variants, technical terms, and new words (GPT-4, COVID-19, etc.). Subwords are the Goldilocks answer: common words get one token; rare words break into recognizable pieces the model can compose. "unprecedented" → ["un", "prec", "edented"].

3. softmax(logits) — why probabilities at all?

After the transformer runs, you have one raw score (logit) per vocabulary token. But why convert them to probabilities? Why doesn't the model just pick a word?

Because language is genuinely uncertain. After "The cat sat on the", there is no single right next token — " mat", " floor", " sofa" are all valid. A probability distribution captures that uncertainty: it says how plausible each option is, rather than committing to one. This lets you make different choices depending on context (safe factual answer vs. creative writing).

For context "The cat sat on the", the model's output after softmax might look like:

" mat"
11.4%
" floor"
7.5%
" sofa"
5.2%
" roof"
3.9%
"…99,996 more…"
72.0%

The remaining 72% is spread across thousands of other plausible tokens. All the numbers add up to exactly 1. This is the full probability distribution over the vocabulary — and it is what the model outputs at every step.

The math behind softmax — how logits become probabilities

The transformer outputs raw scores $z_1, z_2, \ldots, z_V$ (one per vocabulary token, $V \approx 100{,}000$). Softmax converts them:

$$P(\text{token}_i) = \frac{e^{z_i}}{\displaystyle\sum_{j=1}^{V} e^{z_j}}$$

Reading each piece:

$z_i$ The raw logit for token $i$ — any real number.
$e^{z_i}$ Exponential: makes every value positive, since $e^x \gt 0$ for all $x$.
$\sum_j e^{z_j}$ Normalises: dividing by this sum ensures all outputs add to 1.

The higher the logit, the higher the probability. A token whose logit is 5 points above average will dominate the distribution. Softmax preserves ranking while turning arbitrary numbers into a valid probability distribution.

Temperature $\tau$ is a knob that scales the logits before softmax: $z_i / \tau$. At $\tau \to 0$ the highest logit takes all the probability (pure greedy). At $\tau = 1$ you get the model's natural distribution. At $\tau \gt 1$ the distribution flattens — more tokens become plausible, output becomes more random. ChatGPT typically runs around $\tau = 0.7$–$1.0$.

4. sample(probs) — how a token gets picked

You have a probability distribution. Now you need to draw one token from it. There are two main strategies:

Greedy (argmax): always pick the single highest-probability token. Deterministic — run the same prompt twice, get the same answer. Safe for factual tasks, but can produce flat repetitive text.

Sampling: treat the probabilities as a weighted die and roll it. The higher a token's probability, the more likely it is to be chosen — but any token can win, just with varying odds. Run the same prompt twice: you may get different answers. This is how ChatGPT can say "sure!" and also "of course!" to the same greeting.

Analogy: the weighted lottery

Imagine a lottery where " mat" has 11,400 tickets out of 100,000, " floor" has 7,500, " sofa" has 5,200, and the remaining 75,900 tickets are split among every other token. You draw one ticket at random. " mat" wins most often — but " floor" or " sofa" sometimes win too. Greedy would just hand you the ticket with the highest count without any draw.

Temperature changes the ticket counts. Low temperature ($\tau \lt 1$): give " mat" a much larger share, shrink everyone else's. High temperature ($\tau \gt 1$): give everyone a more equal share.

5. tokens.append(next) — the generation loop

Here is the key that makes text generation work: after picking a token, append it to the context and run the model again. Each step, the context grows by one token. Each step, the model conditions on everything that came before — including the tokens it just generated. This self-feeding loop is called autoregression: auto (self) + regression (prediction from prior values).

Concrete trace for prompt "Gravity pulls objects":

1
Input to model
["Gravity", "▁pulls", "▁objects"]
Model outputs distribution — top picks
" downward" 18%,  " toward" 14%,  " together" 7%,  …
Sampled token
" downward"
2
Input to model (prompt + step 1 output)
["Gravity", "▁pulls", "▁objects", "▁downward"]
Model outputs distribution — top picks
"." 42%,  " with" 11%,  " due" 9%,  …
Sampled token
"."
3
Input to model (prompt + steps 1–2)
["Gravity", "▁pulls", "▁objects", "▁downward", "."]
Model outputs distribution — top picks
<END> 71%,  " This" 9%,  " It" 6%,  …
Sampled token → loop ends
<END>

Final response: "Gravity pulls objects downward."

The model sees the entire growing context at each step. That's why the output is coherent: every new token is conditioned on everything that came before it — both the original prompt and all previously generated tokens.

Writing the loop precisely — conditional probability & the chain rule

Now that you understand the loop, here is how to write it in compact mathematical notation.

At step $t$, let $y_1, y_2, \ldots, y_{t-1}$ be all tokens seen so far. The model computes:

$$P(y_t \mid y_1, y_2, \ldots, y_{t-1})$$
$y_t$ The token we're predicting at step $t$.
$y_1 \ldots y_{t-1}$ All previous tokens (prompt + everything generated so far).
$P(\cdot \mid \cdot)$ Conditional probability: "the probability of the left side, given the right side is fixed."

This is just notation for what the pseudocode does: the model reads the whole token list and outputs a probability distribution.

Probability of a whole sequence: the chain rule of probability (you saw this form in JEE: $P(A,B) = P(A) \cdot P(B \mid A)$) extends naturally to T tokens:

$$P(y_1, y_2, \ldots, y_T) = P(y_1) \cdot P(y_2 \mid y_1) \cdot P(y_3 \mid y_1, y_2) \cdots P(y_T \mid y_1, \ldots, y_{T-1})$$

In compact notation using the product symbol $\prod$:

$$P(y_{1:T}) = \prod_{t=1}^{T} P(y_t \mid y_{\lt t})$$

where $y_{\lt t}$ is shorthand for "all tokens before position $t$". This is the probability a trained language model assigns to an entire sequence — it's the product of all the per-step probabilities in the loop.

Why we take the log: products of hundreds of small probabilities (each perhaps 0.05) underflow to zero on a computer. The log turns the product into a safe sum:

$$\log P(y_{1:T}) = \sum_{t=1}^{T} \log P(y_t \mid y_{\lt t})$$

Sums of numbers around $-2$ to $-10$ are numerically safe. This is why every formula in RLHF math uses log-probabilities — not because mathematicians love logs, but because computers need them.

6. What the model is — parameters and the policy

You've now seen what the model does. What is it?

A language model is a function with learnable parameters. The parameters, written $\theta$, are the billions of floating-point weights stored inside the neural network. The function maps a sequence of tokens to a probability distribution over the vocabulary:

$$f_\theta : \underbrace{\text{(sequence of token IDs)}}_{\text{input}} \;\longrightarrow\; \underbrace{P(y_t \mid y_{\lt t};\, \theta)}_{\text{probability distribution over \sim100k tokens}}$$

Pre-training adjusts $\theta$ to make the model good at predicting real text from the internet. Every gradient update is just asking: "which direction do I nudge $\theta$ so that $P(y_t \mid y_{\lt t}; \theta)$ gets higher for the actual next token?" That's the entire training objective.

The nanoGPT codebase by Andrej Karpathy implements exactly this — a working GPT in ~300 lines of Python. The generation loop is literally the pseudocode above. It's a great read if you want to see the real thing with no abstraction.

Why this is the foundation of RLHF — preview

In RLHF, the language model is called the policy — the same word as in reinforcement learning. The "action" the policy takes at each step is sampling a token from $P(y_t \mid y_{\lt t}; \theta)$. The entire goal of RLHF is to adjust $\theta$ so that the policy generates tokens that score highly on human preferences.

When you later see expressions like $\mathbb{E}_{y \sim \pi_\theta}[\, r(y) \,]$ in an RLHF paper — "expected reward under the policy" — the policy $\pi_\theta$ is exactly this function, running the generation loop to produce the sequence $y$.

These are warm-up checks, not the gate. Answer from memory before clicking.

Q1 — What does softmax(logits) return?

Correct. Softmax maps the transformer's raw logits into a valid probability distribution over the full vocabulary at each step.

Softmax doesn't pick — it outputs a distribution. Picking a single token (greedy/sampling) is a separate step applied to that distribution.

Q2 — At step 4 of generation, what is the input to the model?

Exactly. The context grows by one token each step — original prompt plus everything generated so far. That's the "auto" in autoregression: the output feeds back in as input.

Look at line 5 of the pseudocode: tokens.append(next). Every new token gets appended — so by step 4, the model sees the prompt plus tokens 1, 2, and 3.

Q3 (hard) — Greedy decoding picks the highest-probability token at every step. Yet beam search can sometimes find a sequence with higher total probability $P(y_{1:T})$ than greedy. How?

Exactly right — this is a direct consequence of the chain rule. $P(y_1, y_2) = P(y_1) \cdot P(y_2 \mid y_1)$. Greedy maximises $P(y_1)$, but the winning $y_1$ might lead to a terrible $P(y_2 \mid y_1)$. A slightly weaker $y_1$ might open a path where $P(y_2 \mid y_1)$ is dramatically higher, giving a larger product overall. Beam search keeps several "candidate paths" alive to find such globally better sequences. Greedy is locally optimal; the chain rule is why that doesn't guarantee global optimality.

Think about the chain rule: $P(y_{1:T}) = \prod_t P(y_t \mid y_{\lt t})$. Greedy maximises each factor one at a time. But maximising each factor of a product doesn't maximise the product — early choices change what probabilities are available later.
